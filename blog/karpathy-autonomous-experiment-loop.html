Karpathy's Autonomous Experiment Loop

Vincent James Valero ~4 min read

On March 7, Andrej Karpathy pushed a 630-line Python script to GitHub, went to sleep, and woke up to 50 completed ML experiments — no human in the loop. The script, called AutoResearch, had already found a better learning rate and committed the proof to git.

The surface story is about autonomous machine learning research. The deeper story is about a design pattern that applies far beyond ML — and a 40-line Markdown file that makes it all work.

The Problem It Solves

Karpathy was stuck in a tedious manual loop: tweak a hyperparameter, run the model, check the metric, decide whether to keep the change, repeat. AutoResearch automates that entire cycle. A typical session runs ~12 experiments per hour. An overnight run on a single GPU can cover 80–100 experiments — territory that would take a human researcher several working days.

Three Primitives, Not One

The clever part isn't the AI agent — it's the structure underneath it. The loop depends on exactly three things:

  • Editable asset — one file the agent can modify (e.g. train.py). Every change is a reviewable git diff.
  • Scalar metric — one number that determines improvement, computable without human judgment.
  • Time-boxed cycle — a fixed run duration so every experiment is directly comparable.

Each constraint is doing specific engineering work. The fixed time budget makes experiments comparable regardless of what changed. The single file keeps the search space interpretable. The single metric eliminates ambiguity about what "better" means.

Any metric that requires a committee to interpret will be exploited by an autonomous loop running 100 experiments a night. Goodhart's Law applies with full force here.

The Real Star: program.md

The most important file in the repo isn't train.py — it's program.md. This single Markdown document carries three things at once: instructions (what to search), constraints (what must not change), and stopping criteria (when to stop and report).

No other format handles all three cleanly. YAML encodes structure but not reasoning. Python is executable but not legible as a strategy. JSON has no narrative. Markdown sits at the exact intersection of human editability and agent parseability.

Writing a good program.md is the highest-leverage skill in this whole system — not writing the training script, not configuring the agent. The quality of the document is the binding constraint on the quality of the results.

It's Not Just for ML

The pattern generalizes anywhere you have a scriptable asset, a measurable outcome, and a time-boxed evaluation cycle. A few examples:

  • Database query optimization — edit a query config, measure p95 latency on a benchmark
  • Support ticket routing — edit routing rules, measure accuracy on a labeled hold-out set
  • RAG pipeline tuning — edit chunk size and retrieval settings, measure faithfulness score
  • Agent optimization — Harrison Chase already built autoresearch-agents, applying the same loop to LLM agent prompts and architecture

What Changes for Us

The agent eliminates the repetitive execution work — running, waiting, recording, deciding — so that the human's time goes entirely toward decisions that require actual judgment. The researcher's job shifts from running experiments to designing them.

And here's the uncomfortable implication: a researcher who can't articulate what must stay fixed, what should vary, and what "better" means doesn't yet understand their problem well enough to experiment on it usefully. AutoResearch doesn't just automate experiments — it forces clarity.

The gap between "running experiments manually" and "having an agent run them autonomously" is smaller than most teams assume. The primary investment is in document authorship, not infrastructure.
