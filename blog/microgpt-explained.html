How GPT Works in 243 Lines

Vincent James Valero ~6 min read

Andrej Karpathy — one of the founding members of OpenAI — recently published microgpt: a complete, working GPT implementation in 243 lines of pure Python. No PyTorch. No NumPy. Just os, math, and random. It trains on 32,000 baby names and generates new ones that sound real but never existed.

I went through every line. Here's what it actually does, explained for programmers with no AI background.

The same algorithm that powers ChatGPT — just stripped down to the irreducible core. Everything else in modern AI is efficiency on top of this.

The pipeline, step by step

Data flows through seven distinct stages. Each one transforms the input into something the next stage can use.

1 · Raw data
Download names.txt — 32,000 first names, one per line. Shuffle them so training samples randomly, not alphabetically. Each name is one "document."
2 · Tokenizer
Convert text to integers. Every unique character (a–z) gets an ID. A special BOS token (ID 26) marks both the start and end of every name. "emma" becomes [26, 4, 12, 12, 0, 26].
3 · Embeddings
Look up each token ID in a weight table (wte) to get a 16-number vector. Add a position vector (wpe) so the model knows where each character sits in the sequence.
4 · Attention
Each token asks: "which past tokens are relevant to me?" Three learned projections — Query, Key, Value — produce attention weights. A heatmap of who attends to whom. The model learns that double letters often appear in names, for instance.
5 · MLP block
After attention, a small feed-forward network processes each token further. Two linear layers with a ReLU in between. Residual connections add the output back to the input so gradients can flow cleanly during training.
6 · Loss & backprop
The model predicts probabilities for the next character. Loss is -log(probability of the correct answer). Then loss.backward() walks every calculation in reverse, computing how much each weight contributed to the mistake.
7 · Adam update
Nudge every weight in the direction that reduces loss. Adam tracks momentum (smoothing past gradients) and variance (slowing noisy parameters) — smarter than plain gradient descent.

The part that makes it all work: autograd

The most clever piece of the code isn't the attention mechanism — it's the Value class. Every single number in the model is wrapped in a Value object. When you do math with Values, it silently records how each number relates to the others, building a computation graph as a receipt of every operation.

When the model makes a mistake, .backward() traces back through that receipt using the chain rule from calculus, filling in a .grad on every number. That's how the model learns — not magic, just systematic blame assignment all the way down to the raw weights.

This is exactly what PyTorch's autograd does internally. microgpt just does it with plain Python floats instead of GPU tensors.

What "attention" actually means

When predicting the next character after "emm", the model doesn't just look at the last letter. It looks at all previous characters simultaneously and weights their importance. Those weights are learned — the model figures out on its own that the previous "m" is the most relevant context for predicting what follows a double letter.

Four attention heads run in parallel, each potentially specializing in different patterns. Their outputs are concatenated and projected back to the original dimension.

Inference: generating a name

After training, generation is a loop: start with BOS, run the model, get a probability distribution over all 27 possible next tokens, sample one weighted by probability, feed it back in, repeat. Stop when the model predicts BOS again. The temperature parameter (set to 0.5) divides all logits before softmax — lower means more conservative, higher means more creative.

Why this matters

microgpt is slow — processing one scalar at a time in pure Python. A real GPU can process millions in parallel. But the math is identical. The architecture is identical. Every concept in ChatGPT, Claude, or Gemini traces back to these same building blocks: embeddings, attention, MLP blocks, residual connections, and an optimizer nudging weights toward lower loss.

If you can read this code and understand each piece, you understand the core algorithm behind all of modern AI. Karpathy called it an art project. I think it's the best single-file education in machine learning that exists.

Read it. Run it. Change the dataset. The knobs that matter at 243 lines are the same knobs that matter at 175 billion parameters.
