Prompts Are Code: Treat Them That Way

Vincent James Valero ~5 min read

The first time I wrote a system prompt that actually mattered — one running in production, shaping responses for hundreds of users — I stored it in a Google Doc. No version history. No review process. No tests. I just typed it, copied it into an environment variable, and shipped it.

That was a mistake I won't repeat. A prompt is not a configuration detail or a quick instruction. It's logic. It encodes behavior, constraints, and intent — the same way a function does. And like any code that encodes behavior, it needs to be managed like code.

What Makes a Prompt "Code"

Think about what a system prompt actually does. It defines scope — what the model will and won't address. It sets tone and format. It establishes decision branches: if the user asks X, respond with Y framing. It can encode safety rules, persona, and domain knowledge. Change one sentence and the output changes. Delete a paragraph and a whole class of user requests breaks.

That's logic. You wouldn't edit a routing function in a Google Doc and copy-paste it into production. The same standard applies here.

A prompt that runs in production is a production artifact. Treat it accordingly.

The Practices That Actually Help

Over the past year, I've settled on a few habits that have made a real difference:

  • Version control everything. Prompts live in the repo alongside the code that calls them. Every change gets a commit message. git blame on a prompt has saved me hours of debugging.
  • Write a changelog. When a prompt changes, note why. "Tightened scope — model was hallucinating product names" is useful context six months from now.
  • Separate concerns. Keep the system prompt, the few-shot examples, and the user-injected context in distinct, clearly named variables. Mixing them into one blob makes editing treacherous.
  • Test regressions manually. Maintain a short list of "golden inputs" — prompts that should always produce a specific shape of output. Run them before and after any prompt change.
  • Review prompts like PRs. If the prompt change is significant, have someone else read it. You'll catch ambiguities you've stopped seeing.

The Invisible Bug

Prompt bugs are uniquely nasty because they're often probabilistic. The model doesn't crash — it just starts drifting. Responses get slightly off-tone. Edge cases that used to be handled gracefully start slipping through. Users don't file bug reports; they just quietly stop trusting the feature.

Without version history, you can't bisect the problem. You don't know which edit introduced the drift. Was it the sentence you added two weeks ago? The example you removed? The rephrasing of the persona block?

With a proper history, you can compare, roll back, and iterate. It's not perfect — LLM outputs aren't deterministic — but it's a lot better than flying blind.

This Is Still Engineering

There's a tendency to treat prompt writing as a softer, more creative activity — something adjacent to copywriting rather than engineering. That framing is dangerous. The outputs of a poorly managed prompt surface as production incidents, user-facing failures, and silent degradation.

The tools we already use for code — version control, review, testing, documentation — apply here too. They just need to be picked up and applied intentionally.

Your prompt is doing real work. Give it the infrastructure it deserves.
